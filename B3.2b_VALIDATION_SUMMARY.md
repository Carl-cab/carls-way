## Environment Options

### **Option A: Vercel Preview (Recommended) ⭐**

**Why:** Each preview deployment gets isolated database; zero risk to production.

**Setup:**
```bash
git checkout -b test/b3.2b-validation
git push -u origin test/b3.2b-validation
# Vercel auto-deploys preview environment
# Note preview URL: https://<hash>.carloscab74.vercel.app
```

**Verification:**
- Preview URL is separate from production (`https://carloscab74.vercel.app`)
- DATABASE_URL in preview environment points to separate Supabase branch
- Can destroy preview environment after testing

**Advantages:**
- ✓ Completely isolated from production
- ✓ Uses real Vercel/Supabase infrastructure
- ✓ Full deployment pipeline tested
- ✓ Easy cleanup (delete branch)

---

### **Option B: Local Development Database**

**Why:** Fastest iteration, full control.

**Setup:**
```bash
# Start local PostgreSQL
docker run -d --name postgres-test \
  -e POSTGRES_PASSWORD=test \
  -e POSTGRES_DB=manna_test \
  -p 5432:5432 \
  postgres:16

# Set DATABASE_URL
export DATABASE_URL="postgresql://postgres:test@localhost:5432/manna_test"

# Start local server
npm run dev
```

**Verification:**
- DATABASE_URL explicitly points to `localhost:5432` (not production)
- Schema created by `npm run dev` initial boot
- Can inspect database directly via `psql`

**Advantages:**
- ✓ Fastest test iteration
- ✓ Direct database access
- ✓ Easy to inspect logs
- ✓ Clean environment each time

### **Option C: Staging Database Clone**

**Why:** Production-like infrastructure without real data.

## Critical Safety Checks

Before executing ANY validation test:

```bash
# 1. Verify DATABASE_URL is non-production
echo $DATABASE_URL | grep -v "production\|carloscab\|vercel.app"
# ✓ Should have NO output (safe if no match found)

# 2. Verify database is empty or test-only
psql $DATABASE_URL -c "SELECT COUNT(*) FROM users;"
# ✓ Expected: 0 OR all emails start with "test-"

# 3. Verify schema exists
psql $DATABASE_URL -c "\dt public.*" | head
# ✓ Expected: users, transactions, transfer_intents, provider_webhook_events, etc.

# 4. Verify no connection to production
psql $DATABASE_URL -c "SHOW server_version;"
# ✓ Should NOT match production database version/config
```

## Rollback Plan

If ANY test fails:

```bash
# 1. Revert B3.2b commits
git checkout master
git reset --hard HEAD~2  # Reverts both B3.2b commits
# or
git revert 68cfbc6 b30270b  # Reverts via new commits

# 2. Rebuild and test
npm run build && npm run lint

# 3. Push to master
git push -u origin master

# 4. Vercel auto-deploys (production rolls back to B3.1)

# 5. Archive validation report for debugging
cp /tmp/b3.2b_validation_report.md ./FAILED_VALIDATION_$(date +%s).md
git add FAILED_VALIDATION_*.md
git commit -m "Archive: B3.2b validation failed, rolled back"
```

## Production Enablement Criteria

✅ **Approve for production when ALL of the following are true:**

1. ✓ All 8 validation scenarios **pass** in non-production environment
2. ✓ **No bugs** found during testing
3. ✓ **No production data** was modified
4. ✓ **balance_processed_at** correctly prevents duplicate updates
5. ✓ **returned events** skip without prior settlement
6. ✓ **cash_out** transfers are not debited
7. ✓ **add_money** transfers increase balance once
8. ✓ **Invalid data** (currencies, amounts) is rejected
9. ✓ **error_processing_error** is populated on failures
10. ✓ **No ledger entries, notifications, velocity reversals, or provider calls** occur
11. ✓ **Tech lead review** and sign-off obtained
12. ✓ **Validation report** generated and archived

❌ **Do NOT approve for production if:**
- Any scenario fails
- Any bugs found
- Production data modified
- Tech lead has concerns
- Balance mutations appear incorrect
